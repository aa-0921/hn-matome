# 海外テックニュース日本語翻訳・要約サービス調査

> 調査日: 2026-03-25
> 対象: Hacker News・Reddit 等の海外テックニュースを毎日日本語翻訳・要約しているサービス

---

## エグゼクティブサマリー

**結論：既存サービスは複数存在するが、いずれも個人開発・ニッチ規模にとどまっており、本格的な商業サービスはほぼない。**

HN・Redditの日本語翻訳まとめは「個人が作って公開している」段階。ユーザー数・収益・継続性いずれも不明・小規模のものが多く、市場としては未開拓に近い。

## 主要サービスの詳細

### Zeli（最有力競合）

- HN + HuggingFace日次AI論文の両方をカバー
- AI要約でスキミング（読む価値判断）を支援
- PWA・モバイル対応
- Telegramチャンネルも展開
- **唯一「商業サービス感」のある作り**

### Buzzing（規模最大・ただし海外発）

- 中国語・英語・日本語対応の多言語プラットフォーム
- HN以外にReddit・Bloomberg・BBC・WSJなど30+ソース
- リアルタイム更新
- **日本語はあくまで「多言語の一つ」として追加された機能**

### Nook（OSS参考実装）

- Reddit・HN・arXiv・GitHub Trending・技術ブログRSSを統合
- AWS Lambda + S3 + EventBridge で自動化。月平均12円の運用コスト
- Google Gemini API でLLM要約
- セルフホストのみ。公開サービスとしては提供されていない
- **自分でサービスを作る際の参考実装として有用**

## 差別化の空白地帯（機会）

| 観点 | 現状の空白 |
|---|---|
| **Redditに特化した日本語まとめ** | 本格的なものがない。reddit-itnews.comは2ch風で情報密度が低い |
| **ニュースレター形式（メール配信）** | 調査範囲では発見できなかった |
| **複数ソース横断＋キュレーション** | Buzzingは多いが日本語向けに最適化されていない |
| **コメント/議論の翻訳・要約** | catnose.meがやっているが他は少ない |
| **GitHub Trending + HN + Reddit の統合** | Nook（OSS）がやっているが公開サービスはない |
| **通知・パーソナライズ機能** | どのサービスも汎用的。「自分の興味分野だけ」を選べるものがない |

---

## リスクと注意点

- **継続性リスク高**: 個人運営が多く、急停止・更新停止の可能性がある（nunawa.netは「継続を保証しない」と明記）
- **精度の問題**: AI翻訳特有の誤訳・ハルシネーションをほぼ全サービスが免責事項として記載
- **Reddit APIコスト**: 2023年以降 Reddit API 有料化で維持コストが上昇。個人が長期継続しにくい構造
- **Buzzing の存在**: 本格的に参入すると中国発の Buzzing が主要な競合になりうる

---

## 結論・示唆

**「毎日・日本語・テックニュース翻訳」の市場は存在するが、プロダクトとして整備されていない。**

- 完成度が高いのは **Zeli** と **Buzzing** の2つのみ
- Zeliは HN＋AI論文に特化、Buzzingは日本語専用ではない
- **ニュースレター配信・Reddit特化・パーソナライズ** の3軸はほぼ手つかず
- 個人開発で参入余地は十分ある。ただし Reddit API コストと継続運用コストは事前試算が必要

## 開発・収益化の実現可能性調査

> 追記日: 2026-03-25
> 「このサービスを個人開発して稼ぐ場合」の観点で追加調査した結果

### データ取得: API vs スクレイピング

| ソース | 取得方法 | 許可 | コスト |
|--------|---------|------|--------|
| **HackerNews** | 公式Firebase API | 完全OK・制限なし | 無料 |
| **Reddit（個人・非商業）** | 公式OAuth API | OK | 無料（100 QPM） |
| **Reddit（商業利用）** | 公式OAuth API | 有料契約が必要 | $0.24/1,000リクエスト |
| **スクレイピング（どちらも）** | HTML解析 | **ToS違反・NG** | — |

- HN の公式 Firebase API は認証不要・無料・レート制限なし。これが正規の方法
- Reddit のスクレイピングは Developer Terms で明確に禁止
- **稼ぐ目的**でサービスを作る場合、Reddit は商業利用扱いになる可能性が高く、有料契約が必要になりうる

### 需要の実在性

- 既存の無料サービス（Zeli・Buzzing・catnose.me 等）がすでに存在する
- Google翻訳/DeepL のブラウザ拡張で原文のまま読める人が増えており、「翻訳するだけ」では差別化にならない
- **独自の解説・視点・キュレーション**がないと流入は見込めない

### 月額課金ニュースレターの現実性

- 「翻訳＋要約」だけでは有料化は難しい（同じことがユーザー自身でもAIを使えばできる）
- 日経クロステックのニュースレター（単体1,100円/月）が成立しているのは大手ブランド＋専門性があってこそ
- 個人開発で有料化するには「著者の解説・独自の切り口」が必須

### 広告収入 vs サブスクリプション

広告収入モデルの構造：

```
収益 = PV（ページビュー） × 広告単価
```

- **ニュースレター（全文メール）は広告モデルと相性が悪い**: メールで完結すると読者がサイトに来ない → PVが増えない → 広告収入にならない
- **広告収入を目的とするならニュースレターは不要または逆効果**
- 最初はサイトのみ作り、SEOで検索流入を積み上げ、Google AdSense で広告収入を得る構成が最もシンプル

### 完全無料構成の実現可能性（HNのみ）

| コンポーネント | 費用 | 備考 |
|--------------|------|------|
| GitHub Actions（public リポジトリ） | **0円** | 2026年の料金変更も public リポジトリは対象外（明記あり） |
| HN 公式 API | **0円** | 完全無料・無制限 |
| OpenRouter 無料枠 | **0円** | 50 req/日。1日10記事翻訳 = 10リクエスト → 余裕で収まる |
| GitHub Pages（ホスティング） | **0円** | 静的サイトは無料 |
| ドメイン | 年1,000〜2,000円（月約170円） | 取らなければ `xxx.github.io` で0円も可能 |

**実質的な費用はドメイン代のみ。ドメインを取らなければ完全0円で動く。**

OpenRouter の無料枠で使えるモデル（DeepSeek R1 等）は翻訳・要約には十分な品質。有料化しても月数十〜数百円程度。

### 最小構成（広告収入目的）

```
HN公式API → GitHub Actions cron（1日1回） → 静的HTML生成 → GitHub Pages公開 → Google AdSense
```

- ニュースレターは作らない
- サイトにコンテンツを蓄積してSEO流入を狙う
- 運用コスト: ドメインなし0円・ドメインあり月170円程度

## HN API で取得できる情報の範囲

> 追記日: 2026-03-25

### 取得できるもの

| データ | 取得可否 | 備考 |
|--------|---------|------|
| 記事タイトル | ✅ | `title` フィールド |
| 外部リンクURL | ✅ | `url` フィールド（リンク先の本文ではない） |
| スコア（いいね数） | ✅ | `score` フィールド |
| 投稿者名 | ✅ | `by` フィールド |
| 投稿時刻 | ✅ | `time` フィールド |
| コメント数 | ✅ | `descendants` フィールド |
| **コメント本文** | ✅ | `text` フィールド。HN上のコメントはHNのDBに保存されているため取得可能 |
| **リンク先記事の本文** | ❌ | HN APIの範囲外。別途URLをfetchしてHTML解析が必要 |

### 実務上の意味

- **コメント翻訳・要約は完全にHN API内で完結する**
  - コメントはHN自身が保管しているため、公式APIで全文取得できる
  - 「英語圏エンジニアの反応・議論を日本語で読む」機能はAPIだけで実装可能
- **元記事の要約は別処理が必要**
  - `url` を別途fetchしてスクレイピングする必要がある
  - 外部サイトごとに構造が異なる（記事サイト・GitHub・YouTube等）
  - 実装コストが上がる + ブロックされるケースも出る
  - **現実的な最小構成ではタイトル翻訳 + コメント要約のみでも十分な価値が出る**

## Claude Code での開発期間見積もり

> 追記日: 2026-03-25

最小構成（HN API → GitHub Actions → 静的HTML → GitHub Pages）を Claude Code で実装する場合の目安。

| フェーズ | 内容 | 期間目安 |
|---------|------|---------|
| **MVP** | HN API取得・翻訳・静的HTML生成・GitHub Pages公開 | **半日〜1日** |
| **コメント要約追加** | 上位コメントをLLMで日本語要約して表示 | **+半日** |
| **検索・アーカイブ** | 過去記事の検索・日付別閲覧 | **+1〜2日** |
| **モバイル最適化** | レスポンシブCSS・PWA対応 | **+半日** |
| **全部入り** | 上記すべて | **3〜4日** |

### 前提

- Claude Codeが骨格コード（スクリプト・HTML・Actions YAMLファイル）をすべて生成する
- 自分の作業は「要件確認・テスト・デプロイ設定」が中心になる
- **詰まりにくい構成**（外部依存が少ない・デプロイ先がGitHub Pagesのみ）なのでぶれにくい

## 差別化要素の有効性評価

> 追記日: 2026-03-25

提案された3つの差別化軸を既存競合との比較で評価する。

### ① コメント・議論の要約（英語コミュニティの反応を日本語で読める）

**評価: ◎ 最も有効な差別化**

- HN APIでコメント全文が取得できるため実装コストが低い
- 既存競合でコメント要約をやっているのは catnose.me のみ
- 「記事の内容」より「エンジニアがどう反応したか」の方が日本語読者に刺さりやすい
- **翻訳だけでは差別化できないが、コメント要約があれば「ここでしか読めない」になる**

### ② 更新速度・検索性・アーカイブ

**評価: ○ 必要条件・単体では差別化にならない**

- 更新速度は競合がすでに毎日対応しており、頻度を上げても気づきにくい
- **検索性・アーカイブは競合が弱い点**（catnose.me・kj-9 はアーカイブが浅い）
- GitHub Pages + 静的HTMLなら日付別アーカイブは容易に実装できる
- 「1年前のHNの反応を日本語で検索できる」は唯一の存在になれる可能性がある

### ③ モバイル体験の最適化

**評価: △ 必須だが差別化にはならない**

- Zeli・Buzzing はすでにモバイル対応済み
- モバイル対応がない状態でリリースすると離脱率が上がるため実装は必須
- ただしこれ単独でユーザーが選ぶ理由にはならない
- **「他で読めないコンテンツ」があった上でモバイル体験を良くする**という順序が正しい

### まとめ

| 差別化軸 | 有効性 | 実装コスト | 優先度 |
|---------|--------|-----------|--------|
| コメント・議論の日本語要約 | ◎ 最大の差別化 | 低（HN API内） | 最優先 |
| アーカイブ・検索 | ○ 中程度 | 低（静的HTML） | 2番目 |
| モバイル最適化 | △ 必須だが差別化にならない | 低 | MVPで対応 |
| 元記事の要約 | ○ あると良い | 高（外部fetch） | 後回し |

**3軸すべてを組み込めば既存サービスとの明確な差別化になる。ただし実装の出発点はコメント要約。**
